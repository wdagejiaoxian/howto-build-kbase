## 第 1 章：项目介绍

> 本章带你了解什么是知识库构建器，以及它为什么比传统笔记软件更强大。
> 阅读时间：**约 3 分钟**。

---

### 1.1 一句话解释

**知识库构建器**是一个让你把自己的资料变成结构化知识库的工具。你只需要把资料丢进去，AI 会自动帮你整理、分类、建立链接，生成一个可以查询的知识库。

打个比方：
- 传统笔记软件：你需要**手动**整理笔记、分类、打标签
- 知识库构建器：你只管**丢资料**，AI 帮你做所有整理工作

---

### 1.2 核心理念：让 AI 当你的"编辑"

这个方法来自前特斯拉 AI 总监 Andrej Karpathy 提出的个人知识库构建理念。

#### 传统方式 vs Karpathy 方式

| | 传统笔记方式 | Karpathy 方式 |
|---|---|---|
| **谁在整理？** | 你自己 | AI 帮你整理 |
| **你怎么做？** | 手动创建文件夹、写摘要、打标签 | 把资料丢进去，然后消费知识 |
| **累不累？** | 很累，很多人坚持不下去 | 不累，你只管看 |
| **知识会增长吗？** | 不会，除非你手动添加 | 会，AI 自动建立概念之间的链接 |

**一句话总结**：把知识当代码写，让 AI 当你的"编译器"。

---

### 1.3 三层架构

知识库构建器采用**双层目录设计**：

```
你的知识库/
├── 原始资料/          ← 你放原始文件的地方（PDF、文章、笔记等）
│
└── wiki/              ← AI 自动生成的结构化知识库
    ├── summaries/     ← 每份资料的摘要
    ├── concepts/      ← 提取出的概念（技术原理、方法论）
    ├── entities/      ← 提取出的实体（项目、工具、人物）
    └── ...            ← 更多结构化页面
```

**工作流程**：

```
原始资料/  →  AI 读取并理解  →  生成摘要  →  提取概念和实体  →  建立链接  →  wiki/
```

**你只需要做一件事**：把文件丢进 `原始资料/` 文件夹。剩下的全部由 AI 完成。

---

### 1.4 与传统笔记软件的对比

| 功能 | 传统笔记（印象笔记/Notion/有道） | 知识库构建器 |
|------|---|---|
| 整理方式 | 手动创建文件夹、分类 | AI 自动分类 |
| 摘要生成 | 手动写 | AI 自动生成 |
| 概念链接 | 手动建立 | AI 自动建立双向链接 |
| 知识查询 | 搜索关键词 | AI 基于知识库回答你的问题 |
| 知识增长 | 需要你主动添加 | AI 自动发现新连接 |
| 数据归属 | 可能在云端 | 全部在你本地电脑上 |

---

### 1.5 适合谁用？

| 用户类型 | 使用场景 | 举例 |
|---------|---------|------|
| **非技术用户** | 整理读书笔记、课程资料、行业报告 | 学生、产品经理、运营人员 |
| **技术用户** | 整理技术文档、API 文档、踩坑记录 | 程序员、架构师、DevOps |
| **学术研究者** | 整理论文笔记、文献综述、实验记录 | 研究生、博士生、研究员 |
| **团队** | 团队知识沉淀、项目文档管理 | 项目组、部门知识库 |

---

### 1.6 进阶能力

当你的知识库积累到一定规模后，它还可以：

#### 🔍 AI 问答引擎

你可以直接向 AI 提问，AI 会基于你的知识库生成答案。答案可以自动保存回知识库，**越问越聪明**。

#### 🔧 健康检查

AI 可以定期扫描知识库，发现：
- 矛盾的信息（同一个概念在不同页面说法不同）
- 失效的链接（引用了不存在的页面）
- 过时的内容（超过 6 个月未更新）
- 新的连接机会（两个相关概念之间还没有链接）

#### 📊 知识可视化

配合 Obsidian 的图形视图，你可以看到知识之间的连接网络，发现意想不到的关联。

## 第 2 章：快速开始（5 分钟上手）

> 本章带你从零开始，5 分钟内完成第一次知识库使用。
> 操作时间：**约 5 分钟**。

---

### 第 1 步：确认项目结构（30 秒）

打开「知识库构建器」文件夹，确认能看到以下结构：

```
知识库构建器/
├── 教程.md              ← 你正在读的文件
├── 原始资料/            ← 你放资料的地方（空的）
└── wiki/                ← AI 生成的知识库（现在是空的）
```

> **✅ 检查点**：如果两个文件夹都在，跳到下一步。

---

### 第 2 步：打开 AI 工具（30 秒）

打开你支持的 LLM Agent 工具（如 OpenCode、Claude Code、Cursor 等），然后打开「知识库构建器」文件夹作为工作目录。

#### 首次使用需要先配置 Skill

> **⚠️ 重要**：AI 工具不会自动发现 `skills/` 文件夹中的 Skill。首次使用时，你需要按照对应平台的配置指南完成设置。
>
> 💡 **如果已经安装并了解该工具**，直接跳到配置文档中的 **Skill 加载** 章节。

1. 打开 `Agent平台安装与配置/` 文件夹
2. 找到你使用的平台对应的配置文档：
   - 使用 **OpenCode** → 打开 `OpenCode安装与配置.md`
   - 使用 **Claude Code** → 打开 `ClaudeCode安装与配置.md`
3. 按照文档中的步骤完成 Skill 配置

#### 配置完成后

打开你的 AI 工具，确保工作目录是「知识库构建器」文件夹。

**OpenCode 用户**：
- **桌面端**：打开 OpenCode 应用 → 点击「打开文件夹」→ 选择「知识库构建器」文件夹
- **终端/命令行**：在终端中 `cd` 到「知识库构建器」文件夹 → 输入 `opencode`

**Claude Code 用户**：
- **桌面端**：打开 Claude Code 应用 → 选择「打开项目」→ 选择「知识库构建器」文件夹
- **终端/命令行**：在终端中 `cd` 到「知识库构建器」文件夹 → 输入 `claude`

**Cursor 用户**：
- 打开 Cursor → 点击「File」→「Open Folder」→ 选择「知识库构建器」文件夹

**其他工具用户**：
- 在你的工具中找到「打开文件夹」或「Open Folder」选项 → 选择「知识库构建器」文件夹

---

### 第 3 步：初始化知识库（2 分钟）

在 AI 对话中输入以下任意一句话：

```
请帮我初始化知识库
```

或者

```
使用 knowledge-base-builder 初始化我的知识库
```

AI 会向你提出 **8 个问题**，用来全面了解你的需求并生成最贴合你使用场景的分类：

#### 完整问卷

| 问题 | 你的回答示例 |
|------|-------------|
| Q1: 你的身份/角色是什么？ | "我是研究生" / "我是前端开发" / "我是产品经理" |
| Q2: 你主要收集什么领域的资料？ | "机器学习" / "中国历史" / "市场营销" |
| Q3: 你的资料主要是什么格式？ | "PDF 论文" / "网页文章" / "代码片段" / "手写笔记拍照" |
| Q4: 你大概有多少份资料？ | "几十份" / "几百份" / "还在持续收集" |
| Q5: 你希望资料分成几个大类？ | "3-4 个" / "越多越细越好" / "你帮我推荐" |
| Q6: 你平时怎么查找资料？ | "按主题找" / "按时间找" / "按项目找" |
| Q7: 你希望知识库帮你做什么？ | "自动摘要" / "概念链接" / "问答查询" / "全部都要" |
| Q8: 是否有特殊的知识组织习惯？ | "按标签分类" / "需要优先级标记" / "没有" |

> **💡 提示**：
> - 如果不确定怎么回答，直接说**"帮我推荐"**即可，AI 会根据你的情况给出建议
> - 你可以一次性回答所有问题，也可以一个一个回答
> - 回答越详细，生成的分类越贴合你的需求

---

### 第 4 步：确认分类目录（1 分钟）

AI 会根据你的问卷回答，在 `原始资料/` 文件夹下创建分类子目录。

#### 你会看到类似的结果

```
原始资料/
├── 编程语言/
├── 算法/
└── 系统/
```

AI 会问你："这些分类是否符合你的需求？需要调整吗？"

- **如果满意** → 回复"可以"或"开始使用"
- **如果需要调整** → 告诉 AI 你想怎么改（如"加一个'数据库'分类"）

> **✅ 检查点**：打开 `原始资料/` 文件夹，确认 AI 已经创建了你确认的分类子目录。

---

### 第 5 步：投放第一份资料（30 秒）

找一份你想整理的资料（任何格式：`.md`、`.txt`、`.pdf`、`.docx`），把它放入 `原始资料/` 文件夹中。

你可以：
- 放入具体的分类子目录（如 `原始资料/编程语言/`）
- 或者直接放入 `原始资料/` 根目录（AI 会自动分类）

#### 然后告诉 AI

```
我放了一份资料，请处理它
```

或者

```
请处理 原始资料/ 下的新文件
```

---

### 第 6 步：查看生成的 Wiki（30 秒）

AI 处理完成后，打开 `wiki/` 文件夹，你会看到：

```
wiki/
├── summaries/     ← 你的资料摘要
├── concepts/      ← 提取出的概念
├── entities/      ← 提取出的实体
├── index.md       ← 知识库索引
└── log.md         ← 操作日志
```

#### 用 Obsidian 查看（推荐）

1. 打开 Obsidian
2. 点击「打开文件夹」→ 选择 `wiki/` 文件夹
3. 你会看到所有生成的知识页面，包括双向链接和图形视图

#### 不用 Obsidian

直接用任何文本编辑器打开 `wiki/index.md`，即可浏览所有知识页面。

## 第 3 章：详细使用指南

### 3.1 问卷引导流程详解

在第 2 章中，你已经体验了初始化问卷。这里详细说明每个问题的作用。

#### 问题详细说明

| 问题 | 影响什么 | 为什么重要 |
|------|---------|-----------|
| Q1: 身份/角色 | 推荐的知识组织方式 | 学生需要课程分类，开发者需要技术栈分类 |
| Q2: 领域 | 概念页的分类目录 | 决定了 `wiki/concepts/` 下的子目录结构 |
| Q3: 资料格式 | 处理策略 | PDF 需要提取文字，网页需要清理格式 |
| Q4: 资料数量 | 处理时间预估 | 几百份资料需要分批处理 |
| Q5: 分类数量 | 原始资料的子目录结构 | 决定了 `原始资料/` 下的文件夹数量 |
| Q6: 查找习惯 | 索引和搜索策略 | 按主题找需要主题标签，按时间找需要时间排序 |
| Q7: 期望功能 | 工作流优先级 | 如果只要摘要，就不需要生成实体页 |
| Q8: 特殊习惯 | 个性化配置 | 满足你的独特需求 |

#### 如果回答后想修改

随时可以告诉 AI：

```
我想重新回答问卷，调整我的知识库配置
```

AI 会重新向你提问，并根据新回答调整分类结构。

### 3.2 资料投放详解

#### 支持的文件格式

| 格式 | 支持程度 | 说明 |
|------|---------|------|
| `.md`（Markdown） | ✅ 最佳 | 直接处理，无需转换 |
| `.txt`（纯文本） | ✅ 最佳 | 直接处理，无需转换 |
| `.pdf` | ✅ 支持 | AI 会提取文字内容 |
| `.docx` | ✅ 支持 | AI 会提取文字内容 |
| `.html` | ⚠️ 部分支持 | 会去除 HTML 标签，保留文字 |
| 图片（.png/.jpg） | ⚠️ 有限 | 需要 AI 支持图片识别（OCR） |

#### 投放方式

**方式 A：放入具体分类目录（推荐）**

如果你已经知道资料属于哪个分类，直接放入对应的子目录：

```
原始资料/
├── 编程语言/
│   └── Python装饰器笔记.md    ← 你新放入的资料
├── 算法/
└── 系统/
```

**方式 B：放入根目录（AI 自动分类）**

如果你不确定分类，直接放入 `原始资料/` 根目录：

```
原始资料/
├── 新资料1.md                 ← AI 会自动分析并移动
├── 新资料2.pdf
├── 编程语言/
├── 算法/
└── 系统/
```

然后告诉 AI：

```
请帮我分类原始资料根目录下的文件
```

AI 会：
1. 读取每个文件的内容
2. 分析主题
3. 移动到最匹配的分类目录
4. 输出分类报告让你确认

#### 批量投放

如果你有大量资料需要处理：

1. 一次性把所有资料放入 `原始资料/`（可以放在根目录或分类目录）
2. 告诉 AI：

```
我投放了 [数量] 份资料，请分批处理
```

AI 会按批次处理，避免一次性处理过多导致出错。

### 3.3 查看 Wiki 详解

#### Wiki 目录结构

```
wiki/
├── summaries/        ← 每份资料的摘要（对应原始资料中的每个文件）
├── concepts/         ← 提取出的概念（技术原理、方法论等）
│   └── [领域]/       ← 按领域分类的概念页
├── entities/         ← 提取出的实体（项目、工具、人物等）
│   ├── projects/     ← 项目实体
│   ├── tools/        ← 工具实体
│   └── platforms/    ← 平台实体
├── analysis/         ← 问答存档（有价值的问答记录）
├── dev-notes/        ← 开发知识点（使用 dev-extract 提炼）
├── index.md          ← 知识库索引（所有页面的目录）
└── log.md            ← 操作日志（记录每次操作）
```

#### 三种核心页面类型

| 页面类型 | 用途 | 示例 |
|---------|------|------|
| **摘要页** | 一份原始资料的精简总结 | "Python装饰器笔记"的摘要 |
| **概念页** | 一个技术原理或方法论的详细解释 | "什么是装饰器"、"递归原理" |
| **实体页** | 一个具体的事物（项目、工具、人物） | "Django框架"、"VS Code编辑器" |

#### 如何浏览 Wiki

**方法 1：通过 index.md 浏览**

| 操作 | 步骤 | 耗时 |
|------|------|------|

#### 健康检查会做什么

当你要求 AI 运行健康检查时，它会：
1. 检查失效链接（引用了不存在的页面）
2. 检查矛盾信息（同一概念在不同页面说法冲突）
3. 检查孤立页面（没有被任何其他页面引用）
4. 检查过时内容（超过 6 个月未更新）
5. 生成报告列出所有问题和修复建议

## 第 5 章：dev-extract 使用指南

> 如果你在使用 AI 辅助开发，dev-extract 可以自动从你们的对话中提炼出有价值的开发知识点。

---

### 5.1 什么是 dev-extract？

dev-extract 是一个**知识点自动提炼工具**。当你和 AI 一起开发项目时，你们会经历：
- 遇到报错 → 修复
- 踩到陷阱 → 找到正确做法
- 讨论技术方案 → 做出决策
- 选择库/框架 → 对比分析

这些过程非常有价值，但对话结束后很容易被遗忘。dev-extract 的作用就是**自动把这些经验提炼成结构化的知识页面**，存入你的知识库。

#### 打个比方

你和 AI 开发项目就像两个人一起修车：
- 修车过程中你们会讨论"这个零件为什么坏了"、"哪种修法更好"
- 修完后，dev-extract 会把讨论结果写成一份"维修手册"
- 下次遇到同样的问题，直接查手册就行

---

### 5.2 两种运行模式

dev-extract 支持两种模式，根据你的需求选择：

#### 模式 A：知识库联动模式

**知识点汇入你的统一知识库**，与你知识库中的概念页、实体页建立链接。

**适合场景**：
- 你已经有知识库构建器生成的知识库
- 你希望开发知识点和知识库中的概念互相链接
- 你希望所有知识集中在一个地方

**效果示例**：

提炼出的知识点页面中会自动包含：
```markdown
## 相关链接
- [[concepts/Docker网络|Docker网络原理]]
- [[entities/tools/Redis|Redis]]
```

#### 模式 B：独立运行模式

**知识点留在项目本地**，不依赖知识库的概念页和实体页。

**适合场景**：
- 你还没有建立知识库
- 你只想记录开发经验，不想维护完整的知识库
- 你在多个项目中使用，每个项目独立记录

**效果示例**：

提炼出的知识点页面中相关链接章节为纯文本：
```markdown
## 相关概念
- Docker 网络、Redis 配置（关键词，非链接）
```

### 5.3 7 类知识说明

dev-extract 会自动识别以下 7 类知识：

| 类型 | 图标 | 说明 | 示例 |
|------|------|------|------|
| **报错与修复** | 🐛 | 遇到的错误及修复方法 | "Docker 容器启动失败：端口被占用" |
| **踩坑记录** | ⚠️ | 你以为会怎样，实际是怎样的陷阱 | "Python 的浅拷贝导致数据被意外修改" |
| **配置配方** | 📋 | 环境配置步骤 | "Nginx 配置 HTTPS 的完整步骤" |
| **方案决策** | 🔀 | 技术方案的对比和选择 | "选 MySQL 还是 PostgreSQL？" |
| **架构设计** | 🏗️ | 系统架构的设计思路 | "微服务拆分原则" |
| **库/框架选型** | 📦 | 选择哪个库或框架 | "React vs Vue 选型分析" |
| **性能优化** | ⚡ | 性能瓶颈定位和优化方法 | "SQL 查询从 3 秒优化到 50 毫秒" |

> **深度要求**：方案决策、架构设计、库/框架选型、性能优化这四类必须达到 **L3 原理层**——不仅要记录"怎么做"，还要解释"为什么"。

### 5.6 触发方式

#### 手动触发（推荐，最可靠）

在对话中说以下任意关键词：

- "记录一下"
- "提炼"
- "extract"
- "存档"
- "把刚才的知识点保存下来"

AI 会立即扫描本次对话，识别并提炼开发知识点。

> **💡 建议**：养成在开发对话结束前说"记录一下"的习惯，这是**最可靠的触发方式**。

#### 关于"自动触发"的说明

> **⚠️ 重要提示**：dev-extract 的 SKILL.md 中写了"session 结束时自动扫描"，但这个功能**依赖平台支持**。

| 情况 | 是否会自动触发 |
|------|---------------|
| OpenCode 平台 + Skill 正确加载 | ✅ 可能会（取决于平台版本） |
| Claude Code / Cursor / 其他工具 | ❌ 不会 |
| 网页版 ChatGPT / Claude | ❌ 不会 |
| Skill 未正确配置 | ❌ 不会 |

**实际建议**：**不要依赖自动触发**。每次开发对话结束前，手动说"记录一下"，这样最保险。

> **为什么教程还写"自动触发"？**
> 这是 dev-extract 的设计目标——未来平台可能会完善 session-end hook 机制。但目前阶段，手动触发是唯一可靠的方式。

### 7.1 为什么用 Obsidian？

你的知识库是由 Markdown 文件组成的。这意味着**任何文本编辑器都能打开它**。但 Obsidian 提供了其他编辑器没有的能力：

| 能力 | 说明 | 对你的价值 |
|------|------|-----------|
| **双向链接** | 页面 A 链接到页面 B 时，B 会自动记录"谁链接了我" | 发现知识之间的关联 |
| **图形视图** | 用可视化图谱展示所有页面之间的连接 | 一眼看到知识网络的全貌 |
| **本地优先** | 所有文件存在你电脑上，不依赖云端 | 数据完全属于你，隐私安全 |
| **插件生态** | 社区开发了数百个免费插件 | 按需扩展功能 |
| **免费个人使用** | 个人使用完全免费 | 零成本 |

> **不用 Obsidian 可以吗？**
> 完全可以！你的知识库是纯 Markdown 文件，用记事本、VS Code、Typora 都能打开。Obsidian 只是让浏览和探索知识变得更好用。

### 7.4 双向链接：让知识互联互通

#### 什么是双向链接？

假设你有一个概念页叫 `[[Python装饰器]]`，另一个概念页叫 `[[Python函数]]`。

在 `[[Python函数]]` 页面中，你写道：

```markdown
装饰器是一种特殊的函数，详见 [[Python装饰器]]。
```

这时：
- `[[Python函数]]` 链接到了 `[[Python装饰器]]`（正向链接）
- `[[Python装饰器]]` 会自动记录"谁链接了我"（反向链接/Backlinks）

#### 如何创建双向链接？

1. 在编辑模式下，输入 `[[`
2. Obsidian 会自动弹出已有的页面列表
3. 选择你想链接的页面
4. 完成！

#### 如何查看反向链接？

1. 打开任意页面
2. 在右侧面板找到「反向链接」(Backlinks)
3. 展开后可以看到所有链接到当前页面的页面

> ![image-20260516042421338](教程.assets/image-20260516042421338.png)

### 8.1 Lint 健康检查

#### 什么是 Lint？

Lint 是对知识库的全面健康检查。就像杀毒软件扫描电脑一样，Lint 会扫描你的知识库，找出潜在问题。

#### 如何运行

在 AI 对话中输入：

```
请运行知识库健康检查
```

#### Lint 会检查什么？

| 检查项 | 说明 | 问题示例 |
|--------|------|---------|
| **失效链接** | 页面 A 链接到页面 B，但 B 不存在 | `[[Python装饰器]]` 链接不存在 |
| **矛盾信息** | 同一概念在不同页面说法不同 | 页面 A 说"Python 3.8 支持"，页面 B 说"不支持" |
| **孤立页面** | 没有被任何其他页面链接的页面 | 新建后从未被引用 |
| **重复摘要** | 同一份原始资料生成了多份摘要 | 同名文件的不同版本 |
| **过时内容** | 超过 6 个月未更新的页面 | 2024 年初的技术文档 |

#### 如何修复问题

Lint 运行完成后，AI 会生成一份报告，列出所有问题和建议。你可以：

1. **逐条处理**：告诉 AI "请修复失效链接"
2. **批量处理**：告诉 AI "请修复所有报告中的问题"
3. **忽略**：有些问题可能不需要修复（如故意保留的孤立页）

---

### 8.2 Archive 归档

#### 什么内容需要归档？

| 情况 | 示例 |
|------|------|
| 过时技术 | "Python 2.7 迁移指南"（Python 2 已停止维护） |
| 已废弃方案 | "使用 Flask 1.x 的旧方案" |
| 重复内容 | 同一主题的多份摘要 |
| 临时笔记 | "待整理的草稿" |

#### 如何归档

告诉 AI：

```
请运行归档检查，把过时的内容归档
```

AI 会：
1. 扫描所有页面的 `updated` 日期
2. 标记超过 6 个月未更新的页面
3. 将归档页面移动到 `wiki/archive/` 文件夹
4. 在页面顶部添加归档提示
5. 更新 `index.md`，将归档页面移入「已归档」分区

> **归档 ≠ 删除**：归档的内容保留原样，只是被标记为"可能过时"，历史参考价值仍在。

### 8.3 问答存档

#### 自动评估 + 用户决策

当你向 AI 提问时，AI 会**自动判断**答案是否有长期价值。如果答案符合以下任一条件，AI 会在回答末尾询问你是否存档：

| 条件 | 示例 |
|------|------|
| 涉及核心概念或重要原理 | "Python 装饰器的工作原理" |
| 包含可复用的代码/配置 | "Docker Compose 完整配置示例" |
| 解决了常见疑问 | "Git rebase 和 merge 的区别" |
| 有横向对比或深入分析 | "MySQL vs PostgreSQL 选型分析" |

#### 你会看到这样的提示

AI 回答完问题后，如果认为答案有价值：

> [AI 的完整回答...]
>
> 💡 这个答案涉及 **Python 装饰器原理**，我认为有长期价值，是否存档到知识库？

#### 你的选择

| 你的操作 | 结果 |
|---------|------|
| 回复"是"或"存档" | AI 执行存档，创建 `wiki/analysis/[主题].md` |
| 不回复 | 不存档，答案仅保留在对话历史中 |
| 回复"否"或"不用" | 不存档 |

#### 手动存档

如果 AI 没有提示，但你觉得答案有价值：

```
请把这个问答存档
```

AI 会立即执行存档操作。

### 8.4 知识链接优化

#### 发现新连接

知识库积累到一定规模后，AI 可以发现潜在的新链接：

```
请扫描知识库，发现可以新增的链接
```

AI 会：
1. 分析所有页面的内容
2. 找出主题相关但没有链接的页面对
3. 建议新增链接

#### 补全缺失链接

如果 Lint 检查发现大量孤立页面：

```
请为孤立页面补全链接
```

AI 会：
1. 读取孤立页面的内容
2. 在知识库中寻找相关页面
3. 在适当的位置添加链接

### 8.5 批量处理

#### 一次性处理大量资料

如果你有几十甚至上百份资料需要处理：

```
我投放了 50 份资料到原始资料/，请分批处理，每批 10 份
```

AI 会：
1. 按批次读取资料
2. 逐批生成摘要、概念页、实体页
3. 每批完成后向你确认
4. 全部处理完成后输出总结报告

#### 为什么需要分批？

- 避免一次性处理过多资料导致出错
- 每批完成后可以检查结果是否符合预期
- 如果某批有问题，可以及时调整策略

### 8.6 版本管理（Git）

#### 为什么用 Git 管理知识库？

| 好处 | 说明 |
|------|------|
| **历史记录** | 每次修改都有记录，可以回退到任意版本 |
| **备份** | 推送到 GitHub/Gitee 后，云端有备份 |
| **协作** | 多人可以同时编辑知识库，Git 自动合并冲突 |
| **对比** | 可以看到某个页面从一个版本到另一个版本的变化 |

#### 基本操作

**初始化 Git 仓库**：

```bash
cd 知识库构建器
git init
git add .
git commit -m "初始化知识库"
```

**日常提交**：

```bash
git add .
git commit -m "新增 Python 装饰器相关资料"
```

**查看历史**：

```bash
git log --oneline
```

**回退到某个版本**：

```bash
git checkout <commit_hash>
```

#### 推荐工作流

| 频率 | 操作 |
|------|------|
| 每次 AI 处理完资料后 | `git add . && git commit -m "处理: [资料名]"` |
| 每周 | 推送到远程仓库备份 |
| 重大变更前 | 创建分支，避免影响主分支 |

### 基础使用

#### Q1: 支持哪些文件格式？

**原始资料**支持以下格式：
- ✅ **最佳**：`.md`（Markdown）、`.txt`（纯文本）
- ✅ **支持**：`.pdf`、`.docx`、`.html`
- ⚠️ **有限**：`.png`、`.jpg`（需要 AI 支持图片识别/OCR）

**Wiki 输出**：全部为 `.md`（Markdown）格式。

---

#### Q2: 知识库有大小限制吗？

**没有硬性限制**。知识库的大小取决于：
- 你的硬盘空间（全部存在本地）
- AI 工具的上下文窗口（处理大量资料时可能需要分批）

实际使用中，几百份资料（几十 MB）完全没问题。如果资料量非常大（上千份），建议分批处理。

---

#### Q3: 可以用中文吗？

**完全可以**。知识库构建器完全支持中文：
- 原始资料可以是中文
- 生成的 Wiki 页面是中文
- 概念名、实体名都可以是中文
- 双向链接也支持中文：`[[Python装饰器]]`

---

#### Q4: 需要联网吗？

**需要**。AI 处理资料、生成 Wiki 都需要联网。

但知识库文件本身存在你本地，**离线时仍然可以浏览和搜索**已生成的 Wiki 内容。

---

#### Q5: 数据安全吗？隐私如何保障？

**你的数据安全取决于你使用的 AI 平台**：

| 平台 | 数据处理方式 |
|------|-------------|
| OpenCode | 取决于你配置的模型提供商 |
| Claude Code | Anthropic 的隐私政策 |
| 本地模型（如 Ollama） | 数据完全在本地，不上传云端 |

**建议**：如果资料包含敏感信息，建议使用本地模型（如 Ollama）进行处理。

### 知识库管理

#### Q6: 如何备份知识库？

**方法 A：复制文件夹**
直接把整个「知识库构建器」文件夹复制到 U 盘或云盘。

**方法 B：使用 Git（推荐）**
```bash
git add .
git commit -m "备份：2026-05-16"
git push origin main
```

**方法 C：Obsidian Sync**
如果你使用 Obsidian，可以开启官方同步服务自动备份。

---

#### Q7: 可以迁移知识库吗？

**可以**。知识库是纯 Markdown 文件，迁移非常简单：

1. 复制整个 `wiki/` 文件夹到新位置
2. 在新位置打开 Obsidian 或 AI 工具
3. 确保 `AGENTS.md`（OpenCode）或 `CLAUDE.md`（Claude Code）也在同一目录

---

#### Q8: 多人可以共用同一知识库吗？

**可以**。有以下方式：

| 方式 | 说明 |
|------|------|
| **Git 协作** | 多人通过 Git 推送修改，自动合并 |
| **共享文件夹** | 放在 NAS 或云盘中，多人同时访问 |
| **Obsidian Sync** | 团队订阅后共享同一个知识库 |

> **注意**：多人同时编辑同一页面可能产生冲突，建议分工明确。

---

#### Q9: 可以用其他笔记软件替代 Obsidian 吗？

**可以**。知识库是纯 Markdown 文件，以下软件都能打开：

| 软件 | 双向链接 | 图形视图 | 免费 |
|------|---------|---------|------|
| **Obsidian** | ✅ | ✅ | ✅ |
| **Logseq** | ✅ | ✅ | ✅ |
| **Typora** | ❌ | ❌ | ❌（付费） |
| **VS Code** | ❌ | ❌ | ✅ |
| **记事本** | ❌ | ❌ | ✅ |

Obsidian 是推荐选择，因为它的双向链接和图形视图最适合知识库浏览。

---

#### Q10: 如何导出知识库为 PDF？

**方法 A：Obsidian + PDF 插件**
1. 安装「PDF Export」插件
2. 打开想导出的页面 → 导出为 PDF

**方法 B：Obsidian + Marp 插件**
1. 安装 Marp 插件
2. 把笔记转为幻灯片 → 导出为 PDF

**方法 C：打印**
在任何编辑器中打开 `.md` 文件 → 浏览器打开 → `Ctrl + P` 打印为 PDF

### 问题排查

#### Q11: 资料分类不准确怎么办？

告诉 AI：

```
这个资料不应该在"编程语言"分类下，请移到"数据库"分类
```

AI 会移动文件并更新相关索引。

---

#### Q12: Wiki 内容有误如何修正？

告诉 AI：

```
wiki/concepts/Python/装饰器.md 中关于"装饰器性能"的描述有误，正确应该是...
```

AI 会：
1. 读取该页面
2. 修正错误内容
3. 更新 `updated` 日期
4. 记录到 `log.md`

---

#### Q13: 遇到报错如何处理？

| 报错 | 解决方法 |
|------|---------|
| AI 无法读取文件 | 确认文件路径正确，文件格式支持 |
| Wiki 生成空白页面 | 检查原始资料是否为空或格式不支持 |
| 双向链接失效 | 运行 Lint 检查，让 AI 修复失效链接 |
| Skill 未被加载 | 检查平台配置文档，确认 Skill 路径正确 |

如果问题未解决，可以向我们反馈。

### 高级问题

#### Q14: 可以自定义页面模板吗？

**可以**。页面模板位于 `agents/templates/` 目录下：

1. 打开 `agents/templates/` 中对应的模板文件（如 `entity.md`、`concept.md`）
2. 修改模板内容
3. 保存

> **注意**：修改模板会影响 AI 后续生成的所有页面，建议先备份。

---

#### Q15: dev-extract 可以在其他项目中使用吗？

**可以**。dev-extract 支持路径自定义：

1. 在其他项目的根目录创建 `.dev-extract.json`
2. 设置 `wiki_root` 指向你的统一知识库路径
3. 把 dev-extract 的 SKILL.md 加载到该项目的 AI 工具中

这样多个项目的开发知识点都会汇入同一个知识库。
