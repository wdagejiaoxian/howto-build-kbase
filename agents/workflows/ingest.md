## Ingest 工作流

> 处理新源文档的标准流程，支持增量状态和分批执行

---

### 阶段一：启动检查

**读取 `wiki/.ingest-state.md`**

```
检查是否有未完成的批次：
├── 有未完成批次 → 输出「检测到未完成批次，继续中...」
│                  显示：剩余 X 个文件，上次停在哪
│                  用户确认后继续完成
└── 无未完成批次 → 进入阶段二
```

---

### 阶段二：资料自动分类（Step 0）

**扫描原始资料/ 和 原始资料/待分类/ 下的 markdown 文件**

```
1. 扫描所有 markdown 文件（不包括图片/资源）
2. 分析每个文件的内容，提取关键词
3. 匹配已有分类目录（读取原始资料/下的子目录作为分类依据）：
   - AI开发：匹配 AI/LLM/机器学习/深度学习/Agent/LangChain/LangGraph 等关键词
   - 编程语言：匹配 Python/JavaScript/Go/Java 等语言名称
   - Web开发：匹配 Vue/React/Angular/Django/Flask/HTML/CSS 等框架名
   - DevOps：匹配 Docker/K8s/Nginx/CI-CD/部署/域名等关键词
4. 判定置信度并分类：
   - >80% 置信度 → 直接移动到对应分类目录
   - 50-80% 置信度 → 输出待确认列表，等待用户确认
   - <50% 置信度 → 保留在待分类/，提示用户手动分类或新建分类
5. 输出分类报告
```

---

### 阶段三：创建批次

**每批 5 个文件，显示待处理列表**

```
## 待处理文件（共 N 个）

| 文件 | 分类 | 摘要 | 概念 | 实体 |
|------|------|------|------|------|
| A.md | AI开发 | ○ | ○ | ○ |
| B.md | Web开发 | ○ | ○ | ○ |
| C.md | DevOps | ○ | ○ | ○ |
| ... | ... | ... | ... | ... |

（○ = 待完成，✓ = 已完成，? = 进行中）

## 批次规划

每批 5 个文件，本次处理批次 #X
完成后询问：继续下一批还是暂停？

用户确认后 → 进入阶段四
```

---

### 阶段四：批次执行

**分批处理，每批 5 个文件，每步完成后更新 `.ingest-state.md`**

```
批次 #N 开始

### 文件 1：A.md

Step 1: 创建摘要页
  → 读取 原始资料/AI开发/A.md
  → 创建 wiki/summaries/A.md
  → **立即更新 .ingest-state.md**：标记 摘要 ✓
  → 如果失败 → 标记 ?，等待用户处理

Step 2: 创建概念页
  → 检查 wiki/concepts/AI开发/ 是否已有相关概念
  → 创建/更新概念页
  → **立即更新 .ingest-state.md**：标记 概念 ✓

Step 3: 创建实体页
  → 检查 wiki/entities/ 是否已有相关实体（工具/项目/平台）
  → 创建/更新实体页
  → **立即更新 .ingest-state.md**：标记 实体 ✓

### 文件 2-5：重复以上步骤

批次 #N 完成 → 更新 .ingest-state.md：
→ 标记批次状态为「已完成」
→ 更新全局统计
```

---

### 阶段五：批次间交互

```
批次 #N 完成，输出：

## 批次 #N 完成

| 文件 | 摘要 | 概念 | 实体 |
|------|------|------|------|
| A.md | ✓ | ✓ | ✓ |
| B.md | ✓ | ✓ | ○ |
| C.md | ✓ | ○ | ○ |
| ... | ... | ... | ... |

全局进度：
- 摘要页：X/N (Y%)
- 概念页：X/N (Y%)
- 实体页：X/N (Y%)

剩余文件：M 个
继续下一批？（是/暂停/重新规划）

用户选择：
- 是 → 创建新批次，继续阶段四
- 暂停 → 保存状态，结束
- 重新规划 → 重新扫描，调整批次
```

---

### 阶段六：批次全部完成

```
所有批次完成，输出：

## Ingest 完成报告

- 处理文件：N 个
- 创建摘要页：N 个
- 创建概念页：M 个
- 创建实体页：K 个

更新文件：
- wiki/index.md ✓
- wiki/log.md ✓
- wiki/.ingest-state.md ✓（标记全部完成）
- wiki/.lint-state.md ✓（追加待检查页面）
```

### Ingest Prompt 模板

```markdown
## Ingest任务

### 输入
- 源文件: `原始资料/[分类]/[文件名].md`
- 技术领域: [根据内容判定]

### 执行

#### 阶段一：启动检查
1. 读取 `wiki/.ingest-state.md`
2. 检查是否有未完成的批次
3. 如果有，输出状态并询问用户是继续还是重新开始

#### 阶段二：资料自动分类
1. 扫描原始资料/ 和 原始资料/待分类/ 下的所有 markdown 文件
2. 分析每个文件的内容，提取关键词
3. 匹配已有分类目录：
   - AI开发：匹配 AI/LLM/机器学习/深度学习/Agent/LangChain/LangGraph 等
   - 编程语言：匹配 Python/JavaScript/Go/Java 等语言名称
   - Web开发：匹配 Vue/React/Angular/Django/Flask 等框架名
   - DevOps：匹配 Docker/K8s/Nginx/CI-CD/部署/域名 等关键词
4. 判定置信度并分类：
   - >80% → 直接移动到对应分类目录
   - 50-80% → 输出待确认列表，等待用户确认
   - <50% → 保留在待分类/
5. 输出分类报告

#### 阶段三：创建批次
1. 整理待处理文件列表
2. 每批 5 个文件，规划批次
3. 显示批次列表，等待用户确认

#### 阶段四：批次执行（每个文件执行以下步骤）
1. **Step 1: 创建摘要页**
   - 读取源文件内容
   - 创建 `wiki/summaries/[文件名].md`
   - **立即更新 .ingest-state.md**：标记 `摘要 ✓`
2. **Step 2: 创建概念页**
   - 检查 `wiki/concepts/[领域]/` 是否已有相关概念
   - 创建/更新概念页
   - **立即更新 .ingest-state.md**：标记 `概念 ✓`
3. **Step 3: 创建实体页**
   - 检查 `wiki/entities/` 是否已有相关实体
   - 创建/更新实体页
   - **立即更新 .ingest-state.md**：标记 `实体 ✓`

#### 阶段五：批次间交互
1. 输出批次完成报告
2. 显示全局进度
3. 询问用户：继续下一批/暂停/重新规划

#### 阶段六：完成
1. 更新 `wiki/index.md`
2. 更新 `wiki/log.md`
3. 更新 `wiki/.ingest-state.md`：标记全部完成
4. 更新 `wiki/.lint-state.md`：追加待检查页面
5. 输出最终完成报告

### 要求
- 严格使用模板
- frontmatter必须完整
- **每步完成后立即更新 .ingest-state.md**
- **文件名规则**：标题中的空格用连字符替代
  - 例：`Composition API` → 文件名 `Composition-API.md`
  - 链接使用：`[[concepts/领域/Composition-API]]`
- 内部链接使用 [[wiki/path|显示名]] 格式
- 突出可执行的代码/命令
- **不可跳过任何步骤**：摘要→概念→实体，必须按顺序完成
```
