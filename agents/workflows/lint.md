## Lint 工作流

> Wiki 健康检查和优化，支持增量检查

### 核心概念

**增量检查**：基于 wiki/.lint-state.md 的「待检查页面」列表，每次只检查需要检查的页面，避免重复检查和上下文溢出。

**状态流转**：

```
待检查 → 检查中 → 已检查（有问题/无问题）
                  ↓
            修改时 → 待检查（重新移回）
```

---

### 流程步骤

```
Step 1: 读取 lint-state
  → 读取 wiki/.lint-state.md 获取待检查页面列表
  → 实时扫描 wiki/ 目录，比对待检查列表是否与实际文件匹配

Step 2: 分批检查
  → 按分类从「待检查页面」中取出一批（建议每批 5-10 个页面）
  → 如果上下文即将溢出，立即停止，保存当前进度
  → 更新 .lint-state.md，记录「本次已检查」和「剩余待检查」

Step 3: 汇总报告
  → 生成问题列表（按优先级排序）
  → 问题追加到「历史问题记录」
  → 已检查页面从「待检查」移到「已检查」

Step 4: 建议下一步
  → 根据上下文剩余空间，建议继续检查还是下次再查
  → 列出最高优先级待修复问题
```

### Lint Prompt 模板（增量版）

```markdown
## Lint任务 - Wiki增量检查

### 输入
- 本次检查范围：读取 wiki/.lint-state.md 的「待检查页面」列表
- 本次检查目标：检查待检查列表中的 N 个页面
- 每批建议数量：5-10 个页面（根据上下文剩余空间调整）

### 执行

#### 第一步：读取待检查列表
读取 wiki/.lint-state.md 中的「待检查页面」列表，确认本次要检查的页面。

#### 第二步：分批检查
按以下优先级和批次进行检查：

**批次 1（高优先级）**
- 结构验证：
  - frontmatter 完整性：type、tags、created、updated 必填字段
  - source 有效性：summaries 的 sources 指向的原始文件是否存在于 原始资料/
  - 文件名校验：链接路径与实际文件系统是否匹配
- 内容质量：
  - 空内容检查：页面内容少于 100 字
  - 自链接检查：页面自己链接自己的情况

**批次 2（中优先级）**
- 连接质量：
  - 孤立页面检查：无 inbound 链接的页面
  - 失效链接：指向不存在的 wiki 页面
- 信息一致性：
  - 矛盾信息：同一概念在不同页面的说法冲突
  - 过时信息：被新证据推翻的 claims

**批次 3（低优先级）**
- 去重检查：
  - 检查同源文件重复摘要
  - 识别标题相似但非同一来源的摘要
- 相似度分析：
  - 识别主题相关摘要组
  - 评估是否需要创建综合概念页整合
- 归档检查：
  - 检查超过 6 个月未更新的页面
  - 检查 expires 字段过期
  - 评估内容时效性

#### 第三步：更新 lint-state
检查完成后：
1. 将已检查的页面从「待检查」移到「已检查」
2. 将发现的问题追加到「历史问题记录」
3. 更新「最后更新」时间

#### 第四步：检查进度报告
报告本次检查结果：

```markdown
## 本次 Lint 报告

### 检查进度
- 本次检查：N 个页面
- 剩余待检查：M 个页面
- 检查分类：xxx/

### 发现问题统计
| 优先级 | 数量 |
|--------|------|
| 高 | N |
| 中 | N |
| 低 | N |

### 高优先级问题（需尽快修复）
| 页面 | 问题类型 | 问题描述 |
|------|----------|----------|
| xxx.md | frontmatter缺失 | 缺少 type 字段 |

### 本次已检查页面
- [[wiki/concepts/.../页面名]]
- ...

### 下次建议
- 继续检查：xxx 分类，还有 N 个页面待检查
- 或修复本次发现的问题
```

---

### 手动触发命令

| 用户说 | AI 执行 |
|--------|---------|
| "运行 lint" / "检查知识库" | 从待检查列表头部开始检查 |
| "继续 lint" | 从上次中断处继续 |
| "重新检查 xxx 分类" | 将该分类所有页面移回待检查 |
| "重新检查所有页面" | 清空已检查记录，全量重新检查 |
| "查看 lint 进度" | 读取 .lint-state.md，汇报当前状态 |
| "清除 lint 记录" | 清空历史问题记录（保留待检查列表） |

---

### 上下文即将溢出时的处理

当 AI 发现上下文剩余空间不足时：
1. **立即停止**当前批次检查
2. **更新 lint-state**：将未检查的页面保留在待检查列表
3. **生成中间报告**：汇报已发现的问题
4. **提示用户**：下次运行"继续 lint"可从上次中断处恢复

```markdown
## ⚠️ 上下文即将溢出

### 已完成检查
- xxx.md（检查完毕）
- yyy.md（检查完毕）

### 本次发现的问题
| 页面 | 问题 |
|------|------|
| xxx.md | 缺少 frontmatter |

### 剩余待检查（N 个页面）
- aaa.md
- bbb.md
- ...

### 下次执行
输入「继续 lint」从上次中断处继续检查
```

### 每周维护流程

```
Step 1: 运行 lint
  → 输入"运行 lint"或"检查知识库"
  → AI 从待检查列表开始增量检查

Step 2: 修复问题
  → 按报告中的「建议执行顺序」修复问题
  → 修复后，问题会从「历史问题记录」中清除

Step 3: 手动重检已修复的页面
  → 说"重新检查 xxx 页面"将修复后的页面移回待检查
  → 或下次 lint 会自动跳过已检查且无问题的页面
```
